# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project: Codex

An AI-native project built with proven agentic engineering patterns. The folder structure is designed for TDD workflows, full telemetry, and multi-agent collaboration with persistent memory.

## Repository Layout

```
Codex/
├── CLAUDE.md                        # This file — AI guidance
├── README.md                        # Human-facing overview
├── .claude/                         # Claude Code settings & custom commands
│   ├── settings.json                # Permissions, hooks, env vars
│   └── commands/                    # Custom /slash commands
├── .github/workflows/               # CI/CD and telemetry pipelines
├── docs/
│   ├── index/                       # Master catalog — start here when navigating
│   ├── knowledge/                   # Stable facts, references, domain knowledge
│   ├── agents/                      # Agent roles, prompts, capabilities
│   ├── memory/                      # Persistent state between sessions
│   ├── context/                     # Assembled context windows and prompt templates
│   ├── brainstorming/               # Explorations, spikes, hypotheses (non-binding)
│   ├── specs/                       # Formal technical specifications (binding)
│   ├── features/                    # Per-feature design docs + acceptance criteria
│   └── todos/                       # Structured task lists and backlogs
├── logs/
│   ├── telemetry/                   # Structured event logs (JSONL)
│   ├── sessions/                    # Per-session AI run logs
│   └── pipeline/                    # CI/CD run records
├── src/                             # Source code
└── tests/
    ├── unit/
    ├── integration/
    └── e2e/
```

## Navigation

- **What exists?** Check `docs/index/` first — it catalogs every doc, spec, and feature.
- **What should be built?** Read `docs/todos/BACKLOG.md` then the relevant `docs/features/` file.
- **Why was something done?** Check `docs/brainstorming/` for explorations, `docs/specs/` for decisions.
- **What do agents know?** `docs/memory/` holds cross-session state; `docs/context/` has prompt templates.

## Document Lifecycle

| Stage | Location | Status |
|-------|----------|--------|
| Raw idea | `docs/brainstorming/` | Exploratory, non-binding |
| Accepted spec | `docs/specs/` | Binding, versioned |
| Active feature | `docs/features/` | Has acceptance criteria |
| Done task | `docs/todos/` | Archived with date |

## TDD Workflow

Tests live in `tests/` mirroring `src/`:

```
tests/unit/          → fast, isolated, no I/O
tests/integration/   → real dependencies, sandboxed
tests/e2e/           → full system, real environment
```

Always write the test file before the implementation file. File names match: `src/foo/bar.ts` → `tests/unit/foo/bar.test.ts`.

## Telemetry and Logging

Every significant session event should emit a JSONL line to `logs/telemetry/`:

```jsonl
{"ts":"2026-06-22T00:00:00Z","event":"session_start","agent":"claude","task":"feature-x"}
{"ts":"2026-06-22T00:01:00Z","event":"test_run","passed":12,"failed":0}
{"ts":"2026-06-22T00:05:00Z","event":"commit","sha":"abc123","branch":"feature/x"}
```

Session logs go to `logs/sessions/YYYY-MM-DD-<slug>.md` in markdown.

Pipeline run records go to `logs/pipeline/` and are generated by CI.

## Agent Collaboration Rules

- Each agent role is defined in `docs/agents/`. Read your role doc before starting work.
- Cross-agent memory (decisions, learned facts) is written to `docs/memory/`.
- Never mutate another agent's memory file without appending — use append-only sections.
- Context assembled for a task goes in `docs/context/` with the task slug as filename.

## Specs vs. Brainstorming

- `docs/brainstorming/` files are **disposable** — write freely, delete when resolved.
- `docs/specs/` files are **versioned decisions** — never delete, only supersede with a new version.
- A spec must have: Problem, Decision, Rationale, Alternatives Considered, and Status.

## CI Pipeline

Defined in `.github/workflows/`. The pipeline stages are: lint → unit tests → integration tests → e2e tests → telemetry push. A stage failure blocks all downstream stages.

## Memory Persistence

When a session produces a reusable insight (API quirk, architecture decision, domain rule), append it to the appropriate `docs/memory/` file. Format:

```markdown
## YYYY-MM-DD — <title>
<one-paragraph summary>
```
